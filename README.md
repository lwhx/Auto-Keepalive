# 🌐 Cloudflare Workers 自动化保活监控系统 (Auto-Keepalive Pro)

基于 Cloudflare Workers 和 KV 存储构建的轻量级、无服务器 (Serverless) 站点保活与状态监控面板。无需额外部署服务器，利用 Cloudflare 免费额度即可实现分钟级的多站点监控与多渠道告警。

---

## 📖 项目由来与设计初衷

在玩转各类免费 Serverless 应用、自动化签到脚本或监控节点时，经常会遇到由于长时间没有外部流量访问，导致容器被运营商强制挂起或进入深度休眠的问题。本项目的诞生正是为了完美解决这一痛点。

1. **源起与核心价值：高效利用 CF 5 个 Cron 免费额度**
   本项目的最初构想是为了帮我个人的 **69云签到**、**Koyeb 免费节点** 以及 **NodeWarden 后台** 进行定时访问保活。有人可能会疑惑：“69云签到和 NodeWarden 项目本身不是已经自带了定时访问/保活功能吗？”
   本项目的真正价值恰恰就在这里——**Cloudflare 免费账户仅有 5 个 Cron（定时触发器）额度。** 如果你部署的每个项目都去单独配置并占用一个 Cron 触发器，宝贵的额度瞬间就会耗尽。而通过本项目，你可以将所有需要定期访问保活的网址**统统归拢在一起，仅占用 1 个 Cloudflare Cron 额度**，即可完成全自动的多渠道状态轮询和保活，大大节省了宝贵且极其有限的 Cron 资源。

2. **致敬前作：增加管理前台与底层体验重构**
   本项目的 Idea 借鉴了 `wxvxx/Auto-keep-online` 项目。但该项目属于“无前台面板”的设计，用户每次需要增删保活监控网址时，都必须登录 Cloudflare 控制台深入到 Worker 的底层代码或项目后台进行手工干预，体验非常不便。此外，其通知机制也仅支持 Telegram 这一种方式。
   为此，本项目**全面增加了现代化的可视化管理前台（Web UI）**，并且**对后端引擎、存储及多渠道加签逻辑进行了彻底的重构**，让保活网址管理和多平台推送可以一站式无缝配置。

3. **对比 Uptime Kuma：零门槛、零服务器成本与免穿透**
   有人会问：为什么不用更强大的 `louislam/uptime-kuma` 呢？不可否认，Uptime Kuma 是目前极为优秀的监控工具，但它要求用户必须拥有 **Docker 环境或 VPS 服务器**。然而，有相当大的一部分受众手里可能并没有 VPS，或者由于服务器处在内网/NAT 环境下，如何公开访问管理面板还需要折腾 Cloudflare Tunnel、FRP 内网穿透等复杂操作。
   相比之下，本项目**只需要你拥有一个免费的 Cloudflare 账号**，就能实现纯网页环境部署，零基础、不需要懂任何 CLI 命令行，几分钟即可轻松上线且运行极为稳定。当然，如果你已经有了部署完善的 Uptime Kuma 运行环境，完全可以略过本项目。

## ✨ 核心特性

* **⚡ 纯 Serverless 架构**：依托 Cloudflare Workers 运行，数据存储于 Cloudflare KV，极速、稳定且完全免费。
* **🎨 现代化可视化面板**：内置响应式 Web UI，支持深色/浅色主题自由切换，所有增删改查通过页面即可闭环。
* **🔒 基础安全认证**：面板采用 Basic Auth 账号密码保护，保障配置数据安全。
* **🔗 灵活的任务管理**：
    * 支持添加无限量保活任务（受限于 CF 免费额度限制）。
    * 独立配置每个任务的探测频率（分钟级）。
    * **支持在线编辑修改**：支持随时对已有的保活任务进行修改，无需删除重建。
* **📢 强大的多渠道通知路由**：
    * 内置 9 大主流推送平台：Telegram, PushPlus, NotifyX (码达), 钉钉, 飞书, Resend (邮件), Gotify, Ntfy, 以及自定义 Webhook。
    * **深度适配 NotifyX (码达)**：已直接在后端内置官方 `https://www.notifyx.cn/api/v1/send/:key` 接口协议，前端仅需填入 API 密钥 (Key) 即可无缝启用。
    * **高级加签机制**：完美支持钉钉、飞书的 HMAC-SHA256 安全签名校验。
    * **细粒度分配**：支持为每个保活网址配置独立的通知渠道（如：重要节点推送到飞书，签到任务推送到 TG）。
    * **通道一键测试**：新增【测试发送】按钮，在保存前可一键检查通知渠道的 Token、URL 或自定义请求头是否配置正确。
    * **规范化通知模板**：消息通知严格包含【任务名称】、【监控网址】、【发生时间】及【异常详情】等字段。
    * **智能降噪**：任务访问失败时每次提醒，恢复正常后仅进行首次恢复提醒。

## 🚀 部署教程 (图文向导)

整个部署过程都在 Cloudflare 网页控制台中完成，无需任何本地开发环境。

### 第一步：创建 KV 数据库
1. 登录 [Cloudflare 控制台](https://dash.cloudflare.com/)。
2. 在左侧导航栏选择 **Storage & Databases** (存储和数据库) -> **KV**。
3. 点击 **Create a namespace** (创建命名空间)。
4. 输入命名空间名称，例如：`KEEPALIVE_DB`，点击创建并记住该名称。

### 第二步：创建并部署 Worker
1. 在左侧导航栏选择 **Workers & Pages** -> 点击 **Create application** -> 选择 **Create Worker**。
2. 为你的 Worker 起一个名字（如 `auto-keepalive`），点击 **Deploy** (部署)。
3. 部署完成后，点击 **Edit code** (编辑代码)。
4. 清空左侧编辑器中的所有默认代码，将本项目的完整代码粘贴进去。
5. 点击右上角的 **Deploy** (保存并部署)。

### 第三步：配置环境变量与 KV 绑定 (关键)
回到 Worker 的详情页面，进入 **设置** (设置) 选项卡：

1. **绑定 KV 存储**：
   * 找到 **Bindings** (绑定) 或 **KV Namespace Bindings**。
   * 点击 **Add binding** (添加绑定)。
   * **Variable name (变量名) 必须严格填写为**：`DB`
   * **KV namespace** 选择你第一步创建的 `KEEPALIVE_DB`。
   * 保存。

2. **设置登录账号密码**：
   * 在设置页面找到 **Environment Variables** (环境变量)。
   * 点击 **Add variable**。添加以下两个文本变量：
     * 变量 1 名称：`ADMIN_USER`，值填入你想要的登录用户名（例如 `admin`）。
     * 变量 2 名称：`ADMIN_PASS`，值填入你的登录密码（例如 `my_secure_pwd`）。
   * 保存并部署。

### 第四步：设置定时触发器 (Cron Triggers)
为了让保活脚本自动运行，需要配置定时触发器：

1. 在 Worker 的 **Settings** (设置) 页面中，找到 **Triggers** (触发器)。
2. 找到 **Cron Triggers**，点击 **Add Cron Trigger**。
3. 在表达式输入框中输入：`* * * * *` （这代表每 1 分钟唤醒一次 Worker 进行检测）或`*/5 * * * *` （这代表每 5 分钟唤醒一次 Worker 进行检测）。一般每 5 分钟唤醒一次 Worker 检测就够了。
4. 点击添加并保存。

> **💡 提示**：此时你可以通过 Cloudflare 为该 Worker 分配的 `.workers.dev` 域名，或者在 "Triggers" -> "Custom Domains" 中绑定你自己的域名来访问控制面板。

## ⚙️ 使用与配置指南

通过浏览器访问你的 Worker 域名，输入你在环境变量中设置的账号密码即可进入仪表盘。

### 1. 添加通知渠道
进入 **📢 通知渠道管理** 标签页：
1. 选择你需要使用的平台。
2. 填写渠道别名（用于你在分配任务时辨识）。
3. 根据所选平台填写对应凭证：
   * **NotifyX (码达)**: 仅需填写官方分配的 API 密钥 (Key)。
   * **钉钉/飞书**: 填写 Webhook URL。为了安全，强烈建议在钉钉/飞书后台开启“加签”功能，并将密钥填入面板的**加签密钥**框中。
   * **自定义 Webhook**: 可通过 JSON 格式自定义请求头（如 `{"Authorization": "Bearer XXX"}`）。
4. **测试通道**：添加完毕后，点击【🔔 测试发送】按钮。如果发送成功，会提示通过；若失败，会直接回显第三方接口报错，极大地方便了排查。
5. 点击【确认添加渠道】。

### 2. 配置保活任务
进入 **🔗 保活任务管理** 标签页：
1. 填写任务名称与完整的 URL 地址（包含 `http://` 或 `https://`）。
2. 设置 **访问间隔**。例如设置为 `5`，则系统每隔 5 分钟探测一次该网址。
3. 在下方勾选需要接收此任务状态的 **通知渠道**（支持多选，允许分配无限推送渠道）。
4. 点击【确认添加任务】。
5. 对于已经添加的任务，支持随时点击【编辑】调回输入框修改，修改时若变更了通道别名，系统会自动进行**级联更新**同步任务依赖。

### 3. 数据持久化保存
❗ **重要**：在面板中进行任何“添加”、“删除”或“编辑”操作后，面板右上角会出现 `⚠️ 有未保存的更改` 提示。**你必须点击【💾 保存配置到服务器】按钮**，所有配置才会真正写入 Cloudflare KV 数据库并在后台定时任务中生效。

## 常见问题 (FAQ)

**Q: 监控的时间误差是怎么回事？**
A: CF Worker 的 Cron Trigger 最小粒度为 1 分钟。如果你设置间隔为 5 分钟，系统会在时间戳对比满足条件后的下一次分钟级循环中触发。因此会有 0~60 秒的自然误差，属于正常现象。

**Q: 会消耗大量 Cloudflare 免费额度吗？**
A: 系统做了极度优化。KV 存储的写入操作仅在“添加/修改配置”或“检测到任务状态发生改变（正常变异常，或异常变正常）”时触发。常规的定时检测仅消耗极少量的读取请求 (Read)，远低于 Cloudflare KV 每天 100,000 次的免费读取额度。
